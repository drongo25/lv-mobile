# Система диспетчеризации и технического обслуживания оборудования  
## (Telegram как интерфейс для операторов)

Для такой компании подойдет не просто “бот”, а полноценная система диспетчеризации и технического обслуживания оборудования с Telegram как интерфейсом для операторов.

---

## Архитектура системы

- **Telegram-бот** – для сотрудников (операторов)
- **Web-панель** – для руководителя и диспетчера
- **Backend / API** – основная логика
- **База данных** – хранение всей информации
- **Модуль аналитики и прогнозирования** – расход топлива, ТО, риски
- **Геомодуль** – карта оборудования и маршрутов
- **Логирование и контроль действий** – аудит всех операций

---

## Функциональные требования

### 1. Реестр оборудования

Для каждого оборудования хранятся:

- ID оборудования
- Название
- Геолокация
- Тип двигателя
- Объем бака
- Тип топлива
- Средний расход топлива
- График ТО
- Фото оборудования
- Серийный номер
- История обслуживания
- Текущее состояние

**Дополнительные поля:**

- онлайн/оффлайн
- критичность объекта
- часы работы
- последнее ТО
- остаток топлива
- прогноз остановки

---

## Основные процессы

### 2. Заправка топлива

**Действия оператора через Telegram:**

- количество литров
- тип топлива
- фото чека
- фото бака/счетчика
- геолокация
- комментарий

**Что делает система:**

- проверяет геолокацию
- сравнивает с координатами объекта
- фиксирует время
- обновляет остаток топлива
- считает расход

### 3. Контроль масла и фильтров

**Оператор отправляет:**

- состояние масла
- фото щупа
- фото фильтров
- отметку замены
- комментарии

**Система:**

- ведет историю замен
- напоминает о следующем ТО
- прогнозирует замену по моточасам

### 4. Учет моточасов

**Вариант A – вручную**  
Оператор фотографирует счетчик моточасов.

**Вариант B – IoT**  
Подключение:

- CAN-шины
- GSM-контроллера
- датчиков генератора

Данные поступают автоматически.

## Telegram-бот для оператора

### Что получает оператор каждое утро

- список задач
- маршрут
- приоритеты
- просроченные задачи

**Пример:**
- Заправить объект №17
- Проверить масло
- Заменить фильтр
- Проверить уровень топлива

### Что может оператор (через кнопки)

- “Начать задачу”
- “Прибыл”
- “Завершил”
- “Передать задачу”
- “Проблема”
- “Нет доступа”
- “Нужна помощь”

### Передача задачи другому оператору

**Workflow:**

1. Оператор не может выполнить
2. Указывает причину
3. Руководитель подтверждает
4. Задача переходит другому

**Логируется:**  
кто передал → кому → почему → когда

---

## Контроль факта выезда (критически важно)

### Минимальный контроль

1. **Геолокация** – проверка расстояния до объекта (радиус 50–100 м)
2. **Фото оборудования** – с timestamp, GPS metadata, AI-проверка совпадения объекта
3. **Фото замененных фильтров** – исключение фиктивного обслуживания

---

## Дополнительные рекомендации

### 5. QR-коды на оборудовании

- QR-код / NFC-метка на каждом объекте
- Оператор сканирует перед началом работ – уменьшает мошенничество

### 6. Антифрод контроль

Система выявляет:

- слишком быстрые перемещения
- поддельные GPS
- повторные фото
- одинаковые изображения
- “невозможные маршруты”

### 7. AI-анализ фотографий

Автоматическое определение:

- наличие течи масла
- загрязненность фильтра
- уровень топлива по шкале
- наличие оборудования на фото

### 8. Контроль топлива (очень важный модуль)

Система считает:

- средний расход
- сезонность
- расход по типу оборудования
- аномалии
- возможный слив топлива

---

## Прогнозирование топлива

### На текущий момент

- сколько топлива осталось
- на сколько часов хватит
- когда нужна дозаправка

### На месяц вперед

- общий расход
- расход по регионам
- расход по типам техники
- бюджет

### Формула прогнозирования

**Базовая:**  
`расход = средний расход × моточасы`

Можно использовать:
- moving average
- seasonal forecast
- ML-модель

## Web-панель руководителя

### Dashboard

- карта объектов
- статус оборудования
- остаток топлива
- критические объекты
- просроченные ТО
- кто сейчас работает
- KPI операторов

### Карта

- все объекты
- цветовой статус
- маршруты операторов
- история поездок

### Отчеты

**По сотрудникам:**
- сколько задач выполнено
- время выполнения
- опоздания
- отклонения маршрута

**По топливу:**
- расход
- остатки
- подозрительные операции

**По оборудованию:**
- частота поломок
- стоимость обслуживания
- uptime

### Управление сотрудниками

Руководитель может:
- создавать сотрудников
- блокировать
- увольнять
- ограничивать доступ
- назначать роли

### Роли

| Роль | Возможности |
|------|--------------|
| Оператор | Только свои задачи |
| Старший оператор | Может распределять задачи |
| Диспетчер | Полное управление задачами |
| Руководитель | Полный доступ |

## Важное замечание

> Главная ошибка таких проектов — делать “просто бота”.  
> На самом деле нужен:  
> - workflow engine  
> - task management  
> - telemetry platform  
> - audit system  
> - geo tracking  
> - analytics platform  

**Telegram – только интерфейс оператора.**

Система сама:
- строит оптимальный маршрут
- минимизирует расход топлива

- просрочка ТО
- оператор не приехал

## Ключевая ценность системы

Система должна отвечать на **5 вопросов** в любой момент времени:

1. Где оборудование?
2. Сколько топлива осталось?
3. Когда оно остановится?
4. Кто отвечает?
5. Что уже сделано и кем?
